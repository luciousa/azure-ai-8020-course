## The Core Problem: Why Not Let the AI Write SQL?

The most common question when building a natural-language data interface: "Why not just have the AI generate SQL from the user's question and run it?"

**The answer is the approved view catalog pattern.** Here's why direct AI-generated SQL is not used for student data:

| Risk | AI-Generated SQL | Approved View Catalog |
|------|-----------------|----------------------|
| SQL injection | AI may produce exploitable dynamic SQL from user input | Parameterized queries only; no user input in SQL strings |
| Data over-disclosure | AI may generate a query returning individual student rows | Views return only aggregated data or role-scoped summaries |
| Unauthorized access | AI may JOIN tables the user isn't authorized to query | Service account has GRANT SELECT only on approved views |
| Data type of data returned | AI may return SSNs, names, or dates in query results | Views exclude PII columns — only analytical aggregates |
| Auditability | Every AI query is different — hard to audit | Fixed set of approved views — audit is straightforward |
| Stability | SQL generated by AI may be non-deterministic | View logic is code-reviewed and version-controlled |

**FERPA implication:** Unrestricted AI-generated SQL could return individual student records — student education records that may not be disclosed without consent. The approved view catalog prevents this by ensuring the SQL layer returns only aggregates or role-scoped summaries, never individual student rows accessible to unauthorized users.

## The Approved View Catalog Architecture

```
User Question
    │
    ▼
.NET API (Orchestration)
    │
    ├─► AI Search metadata index
    │     Returns: view description, parameters to use
    │
    └─► SQL Server (approved views only)
          │
          ├─── read-only service account (ai_svc_readonly)
          │     GRANT SELECT on vw_* only
          │     DENY SELECT on all base tables
          │
          └─── Approved views (vw_*)
                ├── vw_AttendanceSummaryByStudentAndTerm
                ├── vw_AttendanceSummaryBySchoolAndGrade
                ├── vw_LocalAssessmentResultsBySchoolAndGrade
                ├── vw_StateAssessmentSummaryBySchoolAndGrade
                ├── vw_AssessmentGapBySubgroup
                ├── vw_PerformanceVsBenchmark
                ├── vw_LongitudinalProficiencyTrend
                ├── vw_InterventionStudentSummary
                └── vw_DataQualityFlags
```

The `.NET API` never accepts SQL text from any source — not from the AI response, not from the user question, not from the metadata index. It selects a view function by name from a fixed registry and calls it with typed, validated parameters.

## Designing Views for AI Consumption

### Principle 1: Return Aggregates, Not Rows

A view that returns individual student rows is not AI-safe. Every approved view in the catalog must return **aggregated data**: counts, rates, averages, distributions.

```sql
-- ❌ NOT AI-safe: individual student rows
CREATE VIEW vw_StudentAttendance AS
SELECT
    s.StudentID,
    s.FirstName,       -- PII
    s.LastName,        -- PII
    s.DateOfBirth,     -- PII
    a.DaysPresent,
    a.DaysAbsent
FROM dim_Student s
JOIN fact_Attendance a ON s.StudentKey = a.StudentKey;

-- ✅ AI-safe: aggregated view (no individual students, no PII)
CREATE VIEW vw_AttendanceSummaryBySchoolAndGrade AS
SELECT
    sch.SchoolID,
    sch.SchoolName,
    gl.GradeLevelCode  AS GradeLevel,
    t.TermCode         AS TermID,
    sy.SchoolYear,
    COUNT(DISTINCT e.StudentKey)              AS TotalEnrollment,
    AVG(CAST(a.DaysPresent AS FLOAT) /
        NULLIF(a.DaysEnrolled, 0) * 100)      AS AttendanceRate,
    SUM(CASE WHEN a.IsChronicallyAbsent = 1 THEN 1 ELSE 0 END)
                                              AS ChronicallyAbsentCount,
    AVG(CASE WHEN a.IsChronicallyAbsent = 1 THEN 1.0 ELSE 0.0 END) * 100
                                              AS ChronicallyAbsentRate
FROM fact_Attendance a
JOIN dim_Student     s   ON a.StudentKey = s.StudentKey
JOIN dim_School      sch ON a.SchoolKey  = sch.SchoolKey
JOIN dim_GradeLevel  gl  ON a.GradeLevelKey = gl.GradeLevelKey
JOIN dim_Term        t   ON a.TermKey    = t.TermKey
JOIN dim_SchoolYear  sy  ON t.SchoolYearKey = sy.SchoolYearKey
JOIN fact_Enrollment e   ON e.StudentKey = s.StudentKey
                        AND e.SchoolKey  = sch.SchoolKey
                        AND e.TermKey    = t.TermKey
GROUP BY
    sch.SchoolID, sch.SchoolName,
    gl.GradeLevelCode, t.TermCode, sy.SchoolYear;
```

### Principle 2: Exclude PII Columns

Review every column in every view and ask: "Could this column, alone or in combination with others, identify a specific student?" Remove any column that can.

Columns to always exclude from approved views:
- `StudentID`, `LocalStudentID`, `StateStudentID`
- `FirstName`, `LastName`, `MiddleName`, `PreferredName`
- `DateOfBirth`
- `SSN`, `TaxID`, or any government identifier
- `Address`, `Phone`, `Email`
- `HealthCondition`, `DisabilityCategory` (beyond subgroup aggregate flags)

### Principle 3: Apply Small Cell Suppression

When a group has fewer than 10 students, the results may identify individuals even without direct PII. Apply suppression in the view:

```sql
-- Small cell suppression example in assessment gap view
SELECT
    sch.SchoolID,
    sch.SchoolName,
    sg.SubgroupName,
    -- Suppress when n < 10
    CASE WHEN COUNT(*) < 10 THEN NULL ELSE COUNT(*) END AS StudentCount,
    CASE WHEN COUNT(*) < 10 THEN NULL
         ELSE AVG(CAST(ar.IsAtOrAboveGradeLevel AS FLOAT)) * 100
    END AS ProficiencyRate
FROM fact_LocalAssessmentResult ar
JOIN dim_Student s ON ar.StudentKey = s.StudentKey
JOIN dim_Subgroup sg ON ...
...
GROUP BY sch.SchoolID, sch.SchoolName, sg.SubgroupName
HAVING COUNT(*) >= 1  -- include suppressed rows so caller knows the group exists
```

This is both a FERPA requirement and good practice. The view returns `NULL` for suppressed cells — the AI learns to say "data suppressed (n < 10)" rather than showing a number that could identify students.

## Least-Privilege Service Account

The AI service account must have the minimum permissions needed to run the approved views — nothing more.

```sql
-- Create read-only login (SQL Authentication for lab; Windows Auth / managed identity in production)
CREATE LOGIN ai_svc_readonly WITH PASSWORD = 'StrongPassword123!';
CREATE USER ai_svc_readonly FOR LOGIN ai_svc_readonly;

-- Grant SELECT only on approved views
GRANT SELECT ON dbo.vw_AttendanceSummaryByStudentAndTerm     TO ai_svc_readonly;
GRANT SELECT ON dbo.vw_AttendanceSummaryBySchoolAndGrade     TO ai_svc_readonly;
GRANT SELECT ON dbo.vw_LocalAssessmentResultsBySchoolAndGrade TO ai_svc_readonly;
GRANT SELECT ON dbo.vw_StateAssessmentSummaryBySchoolAndGrade TO ai_svc_readonly;
GRANT SELECT ON dbo.vw_AssessmentGapBySubgroup               TO ai_svc_readonly;
GRANT SELECT ON dbo.vw_PerformanceVsBenchmark                TO ai_svc_readonly;
GRANT SELECT ON dbo.vw_LongitudinalProficiencyTrend          TO ai_svc_readonly;
GRANT SELECT ON dbo.vw_InterventionStudentSummary            TO ai_svc_readonly;
GRANT SELECT ON dbo.vw_DataQualityFlags                      TO ai_svc_readonly;

-- Explicitly DENY access to base tables
DENY SELECT ON dbo.dim_Student          TO ai_svc_readonly;
DENY SELECT ON dbo.dim_Staff            TO ai_svc_readonly;
DENY SELECT ON dbo.fact_Attendance      TO ai_svc_readonly;
DENY SELECT ON dbo.fact_GradeRecord     TO ai_svc_readonly;
DENY SELECT ON dbo.fact_LocalAssessmentResult  TO ai_svc_readonly;
DENY SELECT ON dbo.fact_StateAssessmentResult  TO ai_svc_readonly;
DENY SELECT ON dbo.fact_Enrollment      TO ai_svc_readonly;
DENY SELECT ON dbo.fact_InterventionMonitoring TO ai_svc_readonly;
```

**Test the permissions** by logging in as `ai_svc_readonly` and confirming:
- `SELECT * FROM vw_AttendanceSummaryBySchoolAndGrade` → succeeds
- `SELECT * FROM dim_Student` → fails with permission denied
- `SELECT * FROM fact_Attendance` → fails with permission denied

## Parameterized Queries (SQL Injection Prevention)

Every call to an approved view must use parameterized queries. The view name itself is selected from a fixed registry in code — never from user input or AI output.

### ❌ Vulnerable (never do this)

```python
# NEVER do this — user input or AI output in SQL string
school_id = user_input.get("school_id")  # Could be "'; DROP TABLE dim_Student;--"
query = f"SELECT * FROM vw_AttendanceSummaryBySchoolAndGrade WHERE SchoolID = '{school_id}'"
cursor.execute(query)
```

### ✅ Safe (parameterized)

```python
# Safe: parameter binding — school_id is never part of the SQL string
school_id = validate_school_id(user_input.get("school_id"))  # Type-validate first
query = "SELECT * FROM vw_AttendanceSummaryBySchoolAndGrade WHERE SchoolYear = ? AND SchoolID = ?"
cursor.execute(query, [school_year, school_id])
```

### View Registry Pattern (.NET)

```csharp
// ViewRegistry.cs — the only authorized SQL entry points
public static class ViewRegistry
{
    public static readonly HashSet<string> AuthorizedViews = new()
    {
        "vw_AttendanceSummaryByStudentAndTerm",
        "vw_AttendanceSummaryBySchoolAndGrade",
        "vw_LocalAssessmentResultsBySchoolAndGrade",
        "vw_StateAssessmentSummaryBySchoolAndGrade",
        "vw_AssessmentGapBySubgroup",
        "vw_PerformanceVsBenchmark",
        "vw_LongitudinalProficiencyTrend",
        "vw_InterventionStudentSummary",
        "vw_DataQualityFlags",
    };

    public static bool IsAuthorized(string viewName) =>
        AuthorizedViews.Contains(viewName);
}

// In your data service:
public async Task<List<Dictionary<string, object>>> QueryViewAsync(
    string viewName, QueryParameters parameters)
{
    // Never trust viewName from AI output — validate first
    if (!ViewRegistry.IsAuthorized(viewName))
        throw new UnauthorizedAccessException($"View '{viewName}' is not in the approved catalog.");

    // Parameters are typed and validated, never raw strings
    return viewName switch
    {
        "vw_AttendanceSummaryBySchoolAndGrade" => await QueryAttendanceViewAsync(parameters),
        "vw_LocalAssessmentResultsBySchoolAndGrade" => await QueryLocalAssessmentViewAsync(parameters),
        // ... etc.
        _ => throw new InvalidOperationException($"No handler for view '{viewName}'")
    };
}
```

**Note:** Even the student-level view does not return student names or other PII. The `StudentKey` is an opaque surrogate key. The teacher knows which students they have — they don't need the AI to tell them names.

## What to Aggregate in SQL vs. What to Let the AI Calculate

| Calculation | Where to Do It | Why |
|-------------|---------------|-----|
| Attendance rate (%) | SQL view (`AVG(DaysPresent/DaysEnrolled * 100)`) | Accurate; consistent definition |
| Count of chronically absent students | SQL view (`SUM(CASE WHEN IsChronicallyAbsent = 1...)`) | Exact; no floating point error |
| Proficiency rate (% at Level 3+) | SQL view (`AVG(CASE WHEN Level >= 3 ...)`) | Uses the authoritative threshold |
| Gap from target (-3.8 pp) | Either SQL or AI | Simple subtraction; AI is fine |
| "The attendance rate is lower than..." | AI | Natural language judgment from retrieved numbers |
| "Grade X has the lowest attendance" | AI (with all retrieved rows visible) | Ranking from retrieved data |
| "Attendance has been declining for 3 years" | AI (with trend view result) | Interpretation of retrieved trend |
